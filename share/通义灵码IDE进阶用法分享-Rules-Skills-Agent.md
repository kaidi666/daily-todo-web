# 通义灵码 IDE 进阶用法分享：Rules、Skills、Agent

> 适用对象：已经会用通义灵码基础对话能力，想进一步把 AI 用成稳定生产力的研发同学
>
> 核心目标：从“临时提问”升级到“工程化协作”

---

## 一句话先讲清楚

- `Rules`：告诉通义灵码“这个项目的规矩是什么”
- `Skills`：告诉通义灵码“这类任务应该怎么做”
- `Agent`：把任务委派给智能体，让它自主拆解、执行、验证并交付结果

如果只记一个结论，可以记这个：

```text
Rules 解决“每次都要重新讲规则”
Skills 解决“复杂任务总怕漏步骤”
Agent 解决“任务长、范围大、需要自主推进”
```

---

## 为什么团队里值得推广这三个能力

很多同学对 AI IDE 的使用，还停留在“一问一答”阶段：

- 让它写代码时，每次都要重复说明技术栈、目录结构、命名规范
- 让它做代码评审、补测试、写文档时，输出质量不稳定
- 让它处理跨文件、长链路任务时，需要人一直盯着

通义灵码里，这三个能力正好分别对应三类问题：

| 常见痛点 | 用什么解决 | 结果 |
|---|---|---|
| AI 不懂项目约定 | Rules | 让约束长期生效 |
| AI 做事不成体系 | Skills | 让流程标准化 |
| AI 只能被动跟着走 | Agent | 让任务端到端推进 |

---

## 三者怎么分工

| 能力 | 关注点 | 适合解决的问题 | 典型触发方式 |
|---|---|---|---|
| `Rules` | 约束 | 项目规范、目录约定、技术栈、禁用项 | 自动生效、手动引入、模型决策、指定文件 |
| `Skills` | 流程 | 代码评审、接口设计、日志分析、测试生成 | 自动触发或 `/skill-name` |
| `Agent` | 执行 | 多文件修改、重构、缺陷修复、端到端交付 | 智能体模式 / Quest 模式 |

更直观一点：

```text
Rules = 约束层
Skills = 方法层
Agent = 执行层
```

它们不是替代关系，而是叠加关系：

```text
Rules 先告诉 AI 边界
Skills 再告诉 AI 方法
Agent 最后负责把任务做完
```

## Rules：把项目约定沉淀下来

### 它是什么

通义灵码支持项目专属规则。规则存储在项目下的 `.lingma/rules/` 目录中，只对当前工程生效。

它最适合放这些内容：

- 技术栈和基础架构
- 目录结构约定
- 编码规范和命名规范
- 禁止事项
- 测试、构建、提交流程

### 为什么先从 Rules 开始

因为它投入最低、收益最稳。

只要团队里已经存在“每次 review 都会反复提的要求”，就值得先把它写进 Rules。

比如：

- 禁止直接在组件里写请求逻辑
- API 返回格式统一
- 页面组件、hooks、types 各放哪里
- 测试文件放置规则
- 提交前必须执行哪些命令

### 通义灵码里的常见生效方式

通义灵码规则支持几种常见方式：

- 手动引入：需要时用 `#rule` 选择
- 模型决策：由模型根据场景判断是否应用
- 指定文件生效：只在匹配文件范围内生效
- 始终生效：所有智能会话与相关请求都生效

如果是团队规范，优先考虑“始终生效”或“指定文件生效”。

### Rules 写什么最有效

好的 Rules 有三个特点：

- 具体，能执行，不写空话
- 贴近项目现状，不写理想化口号
- 尽量结构化，方便模型检索和遵循

不太建议写这种话：

```text
请写出高质量、优雅、可维护的代码
```

更建议写成这样：

```markdown
## 前端约定
- 组件使用 TypeScript 函数式组件
- 页面文件放在 `src/pages/`
- 通用组件放在 `src/components/`
- 禁止在组件内部直接请求后端，统一走 `src/services/`

## 接口约定
- 响应格式统一为 `{ code, data, message }`
- 请求异常统一通过 `AppError` 抛出

## 提交前检查
- 运行 `npm run lint`
- 运行 `npm run test`
- 运行 `npm run build`
```

### 推荐的团队级 Rules 模板

```markdown
# 项目规则

## 项目背景
- 这是一个面向内部运营的管理后台
- 前后端分离，前端与服务端分别维护

## 技术栈
- 前端：React + TypeScript
- 样式：Tailwind CSS
- 后端：Node.js + NestJS
- 数据访问：Prisma

## 目录约定
- 页面：`src/pages/`
- 通用组件：`src/components/`
- 接口请求：`src/services/`
- 类型定义：`src/types/`

## 编码规范
- 优先使用函数式组件
- Props 必须定义类型
- 常量使用 `UPPER_SNAKE_CASE`
- 工具函数使用 `camelCase`

## 禁止事项
- 禁止使用 `any`
- 禁止直接修改已有公共组件的对外接口，除非明确说明影响范围
- 禁止提交调试日志

## 测试与校验
- 变更公共逻辑时补充测试
- 交付前执行 `npm run lint && npm run test`
```

### 什么时候说明 Rules 生效了

团队里可以用两个现象判断：

- 同样的提示词，AI 生成的代码更接近团队风格了
- 同类问题在代码评审里被重复指出的次数明显下降了

## Skills：把重复任务做成“可复用能力”

### 它是什么

通义灵码的 Skill 本质上是一段可复用的任务说明，核心文件是 `SKILL.md`。

项目级路径：

```text
.lingma/skills/{skill-name}/SKILL.md
```

用户级路径：

```text
~/.lingma/skills/{skill-name}/SKILL.md
```

同名时，项目级 Skill 会覆盖用户级 Skill。

### 它适合解决什么问题

如果某类任务你已经做了 5 次以上，而且每次流程都差不多，就适合写成 Skill。

典型场景：

- 代码评审
- API 设计与文档生成
- 单测补齐
- 日志分析
- 提交信息生成
- 发布检查清单

### Skill 的价值不在“长”，而在“稳”

很多人第一次写 Skill，会想把所有知识都塞进去。其实更好的做法是：

- 只聚焦一类任务
- 写清楚触发场景
- 写清楚执行步骤
- 写清楚输出格式

这比写一篇很长但边界模糊的说明更有效。

### 一个好用的 Skill 结构

```markdown
---
name: code-review
description: 审查代码质量、安全性、性能和团队规范。在用户要求评审代码、审查 PR、检查潜在风险时使用。
---

# 代码评审 Skill

## 审查范围
- 代码质量
- 安全性
- 性能
- 可维护性

## 执行步骤
1. 先理解本次改动目标
2. 找出高风险变更点
3. 按严重程度输出问题
4. 单独列出建议优化项

## 输出格式
### 严重问题
### 建议改进
### 正向反馈
```

### 在通义灵码里怎么用

有两种常见方式：

- 自动触发：直接描述需求，模型会结合 `description` 判断是否调用
- 手动触发：输入 `/skill-name`

例如：

```text
帮我审查一下这次用户权限相关改动
```

或者：

```text
/code-review
```

### 最适合团队落地的 3 个 Skill

如果你准备给组内推广，最建议先从下面三个开始：

1. `code-review`
2. `gen-tests`
3. `api-docs`

原因很简单：

- 使用频率高
- 价值容易感知
- 输出好坏很容易比较

### 适合分享时展示的 Skill 示例

```markdown
---
name: gen-tests
description: 为函数、组件或接口补充测试。在用户要求补单测、生成测试用例、提高覆盖率时使用。
---

# 测试生成 Skill

## 目标
- 补充正常路径
- 补充边界条件
- 补充异常处理

## 执行步骤
1. 识别被测对象职责
2. 列出关键输入输出
3. 生成测试用例
4. 检查断言是否覆盖核心行为

## 输出要求
- 先说明测试覆盖范围
- 再给出测试代码
- 最后说明仍未覆盖的风险点
```

## OpenSpec：把需求、设计和任务拆解沉淀进仓库

### 它是什么

OpenSpec 是一个面向 AI 编程助手的开源、轻量级 spec-driven development 框架。

如果用一句话概括，它解决的是这个问题：

```text
需求和设计如果只存在聊天记录里，AI 很容易越做越偏；
OpenSpec 则把“要做什么、为什么做、分几步做”沉淀成仓库里的规格资产。
```

它的核心价值不是替代通义灵码，而是给通义灵码补上一层“规格层”。

### 为什么它值得加到这份分享里

因为 `Rules / Skills / Agent` 已经覆盖了约束、流程和执行，但在真实团队协作里，往往还差一层：

- 需求怎么沉淀
- 设计怎么评审
- 任务怎么拆解
- 多轮会话之后，系统当前“为什么这样做”还能不能查得到

OpenSpec 正好补的是这层能力。

### 它和 Rules、Skills、Agent 的关系

可以把它理解成一个额外的“规格层”：

```text
OpenSpec = 规格层
Rules    = 约束层
Skills   = 方法层
Agent    = 执行层
```

它们配合起来会更顺：

- `OpenSpec`：先定义需求、设计、任务拆解
- `Rules`：约束实现时必须遵守的项目规范
- `Skills`：规范某类工作的执行方式
- `Agent`：按规格推进实现、验证并交付

### OpenSpec 在仓库里通常长什么样

OpenSpec 的核心目录一般是：

```text
openspec/
├── specs/                 # 当前生效的系统规格
└── changes/               # 某次需求变更的提案、任务和规格增量
    └── add-user-filter/
        ├── proposal.md
        ├── tasks.md
        ├── design.md
        └── specs/
```

它和“只写一份产品文档”最大的区别在于：

- `specs/` 存当前事实
- `changes/` 存这次改动准备怎么变

这样更适合研发团队持续演进，而不是一次性写文档。

### 在通义灵码里怎么用它，才比较稳

这里需要讲清楚一个边界：

- OpenSpec 不是通义灵码内置功能
- OpenSpec 官网当前列出的“原生 slash command 支持工具”里，没有显式写到通义灵码

所以在通义灵码里，更稳妥的用法不是指望它原生集成，而是把 OpenSpec 当作仓库里的共享规格资产来用。

也就是说，你可以这样配合：

1. 用 OpenSpec 初始化并生成 `openspec/` 目录
2. 把需求提案、设计和任务拆解沉淀进去
3. 在通义灵码里让 AI 读取这些规格文件
4. 再结合 Rules、Skills 和 Agent 执行实现

### 如果团队想试跑，可以这样开始

OpenSpec 官方的基础启动方式很简单：

```bash
npm install -g @fission-ai/openspec@latest
openspec --version
openspec init
```

初始化之后，仓库里会生成 `openspec/` 目录，并写入共享的工作流说明。

在通义灵码里，建议直接用自然语言驱动，而不是依赖原生命令，例如：

```text
请基于 openspec/project.md 和 openspec/specs/ 理解当前系统规格。
然后为“订单列表新增渠道筛选”创建一份变更提案，输出 proposal、tasks 和需要修改的 spec。
```

等规格确认后，再继续：

```text
请根据 openspec/changes/add-order-channel-filter/ 下的任务继续实现代码，并在完成后回写任务状态。
```

### 一个很适合分享时讲的组合方式

假设我们要做“用户列表新增角色筛选”。

可以按这条链路来：

#### 第一步：用 OpenSpec 固化需求和任务

在 `openspec/changes/add-user-role-filter/` 下沉淀：

- `proposal.md`：为什么要加这个能力
- `design.md`：前后端怎么配合
- `tasks.md`：页面、接口、类型、测试分别要做什么
- `specs/.../spec.md`：系统行为会新增或修改什么

#### 第二步：让 Rules 约束实现方式

例如：

- 请求统一走 `src/services/`
- 响应结构固定为 `{ code, data, message }`
- 变更公共逻辑必须补测试

#### 第三步：让 Skills 规范具体工作

例如：

- `gen-tests` 负责补测试
- `api-docs` 负责补接口文档
- `code-review` 负责交付前审查

#### 第四步：交给 Agent 推进交付

这时给 Agent 的提示词就可以更具体：

```text
请基于 openspec/changes/add-user-role-filter/ 下的 proposal、design 和 tasks 完成实现。
遵守当前项目 Rules。
完成前端筛选、后端查询接口、类型定义和测试补充。
完成后请说明改动文件、验证结果，以及还有哪些任务未完成。
```

这样做的好处是：

- 需求不只存在聊天记录里
- AI 执行依据更稳定
- 团队成员更容易复盘“这次到底改了什么”
- 后续维护的人也更容易理解历史意图

### 什么时候值得引入 OpenSpec

如果团队已经出现这些情况，就很值得试：

- 一个需求经常跨多人、多轮会话推进
- 需求改动频繁，聊天记录很快失去可追踪性
- 做完代码后，大家说不清“原始需求到底是什么”
- 想把需求、设计、任务和实现串成一个完整链路

### 什么时候没必要急着上 OpenSpec

如果当前还是这些场景，可以先不引入：

- 小需求，当天就能做完
- 只有一个人快速试验
- 还没有稳定的需求评审或任务拆解习惯

这种情况下，先把 `Rules + Skills + Agent` 用顺，通常收益更直接。

### 给组内的推荐说法

分享时可以把它定义成：

```text
OpenSpec 不是多一个“更强的 AI”，
而是多一层“让 AI 和人都能对齐需求”的规格基础设施。
```

## Agent：把复杂任务交给智能体推进

### 先统一一下本文里的叫法

在通义灵码语境里，分享时统一用 `Agent` 更自然。

对团队来说，可以把它理解为：

- 在智能会话里使用“智能体模式”处理工程任务
- 在 Quest 模式里把任务交给智能体自主推进

分享时重点放在“它会自己拆任务、执行和验证”即可。

### Agent 适合什么任务

适合这几类：

- 需要跨多个文件修改
- 需要先分析再执行
- 需要调用终端、读写文件、检查结果
- 需要一段时间持续推进

例如：

- “把用户模块的请求层统一替换成新的 SDK”
- “修复这个构建失败问题并补上回归测试”
- “给订单页新增筛选条件并同步更新接口与页面”

### 通义灵码里 Agent 的两个典型使用入口

#### 1. 智能会话里的智能体模式

适合中等复杂度任务：

- 需要看工程上下文
- 需要修改多个文件
- 需要迭代几轮完成

#### 2. Quest 模式

适合更长链路、更强调交付的任务：

- 复杂功能开发
- Bug 修复
- 代码重构
- 从需求、方案到验收的完整闭环

Quest 的核心特点可以概括成四个词：

- 澄清
- 规划
- 执行
- 验证

如果你分享时只给一句解释，可以这样说：

```text
普通对话像“你一步步带着 AI 做”
Agent / Quest 更像“你把目标交给 AI，让它自己推进到可交付状态”
```

### 什么时候不要上来就用 Agent

下面这些场景，先别急着进 Agent：

- 只是想问一个知识点
- 只需要解释一段代码
- 只是要一个函数示例
- 任务边界还完全没想清楚

这种时候先用普通问答，等目标清晰了再切 Agent，效率更高。

### 给 Agent 下任务，什么提示词更容易成功

高质量任务描述通常包含 4 个要素：

- 目标
- 范围
- 约束
- 验收标准

比如：

```text
请在用户管理模块新增“按角色筛选”功能。
范围包括前端列表页、筛选表单、后端查询接口和类型定义。
保持现有接口返回结构不变，遵守项目 Rules。
完成后请自行检查构建和测试，并说明改动文件与验证结果。
```

这类描述明显比“帮我改一下用户列表”更适合 Agent 发挥。

### 分享时推荐讲的 Agent 价值

别只强调“更聪明”，更值得讲的是这几点：

- 它能自动感知工程上下文，不用你一遍遍补文件背景
- 它能处理多文件、多步骤任务
- 它会带着验证意识去交付，而不是只吐一段代码
- Quest 模式更适合长任务和可追踪交付

## 三者怎么组合，才是通义灵码的正确打开方式

最实用的组合，不是三选一，而是分层配合。

### 组合方式一：Rules + Skills

适合团队规范落地。

例子：

- Rules 里规定接口返回格式、目录结构、测试要求
- `code-review` Skill 负责按团队标准评审改动

结果是：

- 约束是长期的
- 评审是结构化的

### 组合方式二：Rules + Agent

适合真实开发任务。

例子：

- Rules 先告诉 Agent 项目栈和禁用项
- Agent 再去做跨文件实现和验证

结果是：

- AI 不容易“写对了功能，写错了风格”

### 组合方式三：Rules + Skills + Agent

这是最接近团队生产环境的组合。

例子：

- Rules 约束项目标准
- Skill 提供领域流程
- Agent 负责执行落地

可以把它理解为：

```text
Rules 定边界
Skills 给方法
Agent 负责交付
```

### 组合方式四：OpenSpec + Rules + Skills + Agent

如果团队已经进入“需求复杂、跨人协作、需要可追踪”的阶段，这会是更完整的组合。

可以这样理解：

- `OpenSpec` 管需求、设计、任务拆解
- `Rules` 管项目实现边界
- `Skills` 管执行套路
- `Agent` 管端到端落地

这个组合尤其适合：

- 中大型需求
- 多人协作开发
- 功能需要多轮迭代
- 希望把需求和实现一起沉淀在仓库里

一句话记忆：

```text
OpenSpec 负责“做什么”
Rules 负责“别做错”
Skills 负责“怎么做更稳”
Agent 负责“把它做完”
```

## 一个适合组内分享的完整案例

### 场景：给已有后台系统增加“用户状态筛选”

如果只用普通对话，大家很容易这样工作：

1. 先让 AI 改前端页面
2. 再补一句让它改接口
3. 再补一句让它补类型
4. 再补一句让它写测试
5. 最后再问它有没有漏文档

这样的问题是：

- 每一步都要重新补上下文
- 很容易漏掉测试或校验
- 改动范围一大就开始失控

更推荐的做法是：

### 第一步：先写 Rules

把这些约定固化：

- 列表页统一放在 `src/pages/`
- 请求统一走 `src/services/`
- 接口响应格式固定
- 变更公共逻辑时必须补测试

### 第二步：准备 Skill

比如准备一个 `add-feature` Skill，约定新功能开发的标准流程：

1. 分析影响范围
2. 修改页面
3. 修改接口
4. 修改类型
5. 补测试
6. 输出改动说明与验证结果

### 第三步：用 Agent 执行

任务描述可以这样写：

```text
请在用户管理页面增加“用户状态筛选”功能。
需要同时修改前端筛选区、列表查询参数、后端查询接口和相关类型定义。
遵守当前项目 Rules。
完成后请自行检查相关测试和构建结果，并总结改动文件。
```

这时候通义灵码更容易做到的是：

- 先理解范围
- 再拆解任务
- 然后分步修改
- 最后输出验证结果

这就是“工程化使用 AI”的核心差异。

## 组内推广时，建议这样讲

如果分享时间不长，我建议按这个节奏讲：

### 1. 先讲痛点，不先讲概念

先问大家三个问题：

- 有没有总在重复告诉 AI 技术栈和规范
- 有没有让 AI 补代码时经常漏测试、漏文档
- 有没有觉得 AI 一旦任务变长就需要一直盯着

这时候再引出 `Rules / Skills / Agent`，大家更容易代入。

### 2. 每个概念只讲“定义 + 一个例子”

不要一上来讲太多抽象设计。

建议这样配例子：

- `Rules`：项目规范
- `Skills`：代码评审
- `Agent`：多文件功能开发

### 3. 多讲“什么时候用”，少讲“概念边界争论”

组内同学最关心的是：

- 我现在这件事该不该写 Rules
- 这类工作值不值得抽成 Skill
- 什么时候该切到 Agent / Quest

所以比起讲很多定义，更值得给判断标准。

### 4. 最后给一份当天就能试的清单

- 在项目里补 3 条最重要的 Rules
- 建一个 `code-review` Skill
- 选一个真实需求，用 Agent 跑一遍

这样分享结束后，团队更容易真的用起来。

## 一页版结论

| 想解决的问题 | 优先用什么 |
|---|---|
| AI 总是不懂项目规矩 | `Rules` |
| AI 做事不成体系 | `Skills` |
| AI 很难独立推进复杂任务 | `Agent` |
| 需求、设计和任务需要长期沉淀并可追踪 | `OpenSpec` |

最推荐的落地顺序是：

```text
先写 Rules
再沉淀高频 Skills
最后把复杂任务交给 Agent
```

这是因为：

- `Rules` 最容易开始，见效最快
- `Skills` 最容易复制团队经验
- `Agent` 最能放大前两者的价值
